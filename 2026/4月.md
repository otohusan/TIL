## BASE64 と BASE64URL

### BASE64
- バイナリデータを ASCII 文字列に変換するエンコード方式
- `A-Z`, `a-z`, `0-9`, `+`, `/` の 64 文字と、パディング用の `=` を使う
- 3 バイト（24 ビット）を 6 ビットずつ 4 文字に変換するため、元データより約 33% サイズが増える
- メールの添付ファイル（MIME）や、HTML に画像を埋め込む Data URI などで使われる

### BASE64URL
- BASE64 の亜種で、URL やファイル名で安全に使えるようにしたもの
- BASE64 との違いは以下の 2 点：
  - `+` → `-`、`/` → `_` に置き換え（URL のクエリパラメータ等で予約文字と衝突しないように）
  - パディングの `=` を省略することが多い
- JWT（JSON Web Token）のヘッダー・ペイロード部分のエンコードに使われている

### まとめ

| | BASE64 | BASE64URL |
|---|---|---|
| 特殊文字 | `+`, `/` | `-`, `_` |
| パディング | `=` あり | 省略されることが多い |
| 主な用途 | MIME、Data URI | JWT、URL パラメータ |

## 同一オリジンポリシーとCORS
### 同一オリジンポリシー（Same-Origin Policy, SOP）
#### 何か
あるサイト（オリジン）によって読み込まれた文書やスクリプトから、他のサイトにあるリソースにアクセスできる方法を制限するもの（[mdn](https://developer.mozilla.org/ja/docs/Web/Security/Defenses/Same-origin_policy)）

#### 利用される流れ
```
[ユーザー]
  ├─ サイトAにログイン済み
  └─ warui.com を開く

[warui.com の JavaScript]
        |
        | fetch("https://site-a.com/api/me")
        v
[ブラウザ]
        |
        | site-a.com 向け Cookie を付けることがある
        | (Cookie 側の条件次第)
        v
[site-a.com]
        |
        | cookie から認証 OK 判定
        | ログイン済みユーザーの情報を返す
        v
[ブラウザ]
        |
        | ← ここで 同一オリジンポリシー が防ぐ
        | warui.com にはレスポンス本文を渡さない
        v
[warui.com]
  └─ 情報を盗み見できない
```

こんな感じで、同一オリジンポリシーは、他のサイトにあるリソースにアクセスできる方法を制限している

### オリジン間リソース共有(Cross-Origin Resource Sharing, CORS) 
#### 何か
同一オリジンポリシー の制限を、サーバ側が明示的に一部だけ緩める仕組み

#### CORS を使用した場合
```
[ユーザー]
  ├─ サイトAにログイン済み
  └─ warui.com を開く

[warui.com の JavaScript]
        |
        | fetch("https://site-a.com/api/me")
        v
[ブラウザ]
        |
        | site-a.com 向け Cookie を付けて送ることがある
        | (条件次第)
        v
[site-a.com]
        |
        | cookie から認証 OK 判定 ＋ ログイン済みユーザーの情報を返す
        | 
        | ⬇️ の ヘッダーを追加
        | Access-Control-Allow-Origin: https://warui.com
        v
[ブラウザ]
        |
        | ← ⬆️ のヘッダーに登録されているサイト（オリジン）は閲覧を例外的に許可
        | warui.com にレスポンス本文を渡す
        v
[warui.com]
  └─ 情報を閲覧することができる
```

こんな感じで、ブラウザに特定のサイトに対しては、情報を開示していいよって、サーバー側で指定できるようにする仕組みが CORS
